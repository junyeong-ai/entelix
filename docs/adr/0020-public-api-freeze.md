# ADR 0020 — Public API freeze policy (toward 1.0)

**Status**: Accepted
**Date**: 2026-04-27
**Closes**: Phase 6 §"cargo public-api freeze" task. Establishes the
mechanism and policy by which the entelix public surface is frozen
ahead of 1.0 — including how (and when) to deliberately break the
freeze, who reviews the diff, and what the CI gate does.

## Context

Pre-1.0 invariant 12 says we don't ship backwards-compatibility
shims; changes to a name, type, or signature *delete* the old in
the same PR. That rule is fine while every consumer is in-tree.
Once 1.0 ships and external consumers depend on the surface, the
same mindset becomes hostile: an accidental rename or new public
type that we didn't intend to ship is a breaking change, regardless
of whether the test suite still passes.

`cargo-public-api` solves the *visibility* half: it lists every
publicly-reachable item the workspace exposes, in a stable diffable
text format. What it can't do alone is enforce intent — that's what
this ADR is for.

## Decision

### Mechanism

Per-crate baseline snapshots live under `docs/public-api/<crate>.txt`,
generated by `scripts/freeze-public-api.sh`. CI runs
`scripts/check-public-api.sh` which:

1. Builds the live public surface via `cargo public-api -p <crate> --simplified`.
2. Diffs against the committed baseline.
3. Fails on any drift, naming the affected crates and showing the
   unified diff.

The simplified format (`--simplified`) drops auto-trait `Send` /
`Sync` / `Unpin` impls that change spuriously when private types
move. Only meaningful surface changes — added items, removed items,
signature shifts — show up as drift.

The check is a **soft gate** locally (skips with a warning when
`cargo-public-api` isn't installed) so contributors don't need the
tool to run the standard `cargo test`. CI installs it deliberately
via `cargo install cargo-public-api` and **enforces**.

### Policy

Three classes of public-surface change and their handling:

1. **Bug fix that doesn't touch the surface** (most changes). No
   refreeze needed — `check-public-api.sh` passes.
2. **Intentional surface change pre-1.0** (current era). Open the
   PR, let CI fail, then run `scripts/freeze-public-api.sh <crate>`
   locally and commit the updated baseline alongside the change.
   Reviewer reads the baseline diff with the same care as the code
   diff. Invariant 12 still applies — no deprecated shims.
3. **Intentional surface change post-1.0**. New ADR required
   describing the change, its motivation, and its semver impact
   (minor for additions, major for removals/renames). The baseline
   refreeze and the ADR land in the same PR.

This freeze is deliberately mechanical: it doesn't decide *whether*
a change is acceptable, only that it must be **observed and signed
off**. Combined with ADR-0010 (naming taxonomy) and the existing
review checklist, accidental drift is caught at PR time.

### What's in the baseline

Every workspace member except the `entelix` facade crate has a
baseline. The facade is intentionally excluded — its surface is
just `pub use` re-exports of the sub-crates, so its baseline would
double-count every type and produce noisy diffs whenever a
sub-crate adds a re-export. Reviewers of facade-touching PRs lean
on the underlying crate baselines instead.

`entelix-core` dominates by a factor of ~3 — expected, since it
houses the IR types (`ModelRequest`, `Message`, `ContentPart`, …),
the codec / transport traits, and most of the public type vocabulary
the SDK exposes.

## Consequences

✅ Accidental new public items show up at PR time, not at release.
✅ Reviewers see the surface diff inline with code changes.
✅ `cargo install cargo-public-api` is the only tooling addition.
✅ Pre-1.0 surface churn remains cheap (refreeze in one command).
❌ Refreeze adds ~30s of `cargo build` time per affected crate.
❌ Auto-trait drift (e.g. `Send` on private internals) might still
   surface as a non-functional diff if `--simplified` misses
   something — accept the noise; don't gate on it.

## Migration to 1.0

The baseline frozen by this ADR represents the **post-Phase-5
surface**. Phase 6 still has open work that may add or remove
public items (e.g. AgentService HookRegistry plumbing, migration
guide changes, example shuffling). Each such PR refreezes whatever
crate it touches, with the diff visible in review.

When 1.0 lands, the baseline at that point becomes the *frozen*
1.0 surface. Post-1.0, every refreeze pairs with an ADR.
