# ADR 0064 — 1.0 release charter

**Status**: Accepted
**Date**: 2026-05-02
**Decision**: 1.0 candidate release prep — semver promise, public-API enforcement, invariant freeze

## Context

The workspace has shipped 78 RC-unit slices across nine
architectural phases (Phase 1 composition spine through Phase 9
companion-perf hardening / parity completion / invariant audit
sweep). 66 ADRs (0001 through 0066) document every architectural
decision; per-crate `docs/public-api/<crate>.txt` baselines pin
the surface of every workspace member; 18 invariants govern code
that compiles. The 7-차원 audit closed every flaw register entry
(F1-F12) introduced earlier in the design.

What remains undefined is the *contract* a `1.0.0` release puts
on the surface. Without a charter, "1.0" is a number — operators
have no guarantee about what they can rely on across point /
minor / major bumps, and contributors have no rule for when a
change requires a new ADR vs. an in-PR delete.

This ADR fixes the contract.

## Decision

### Versioning — strict semver, lockstep workspace

The workspace ships in lockstep: every crate carries the same
version on crates.io. Single number, single decision per
release. Mixed versions (`entelix-core 1.2`, `entelix-mcp 1.1`)
are forbidden — they would let a downstream user combine an
incompatible pair and discover the break at runtime.

- **MAJOR** (`X.0.0`) — any breaking change to the public surface
  enumerated in `docs/public-api/<crate>.txt`, any removal /
  rename / signature change, any change to the 18 invariants.
  Post-1.0, every major bump requires a dedicated ADR explaining
  the migration path the operator owes themselves.
- **MINOR** (`1.X.0`) — additive surface only. New trait method
  with default impl, new `*Builder::with_*`, new companion crate,
  new ADR landing the addition. Per-crate baseline grows but
  never shrinks.
- **PATCH** (`1.0.X`) — internal fixes / bug fixes / doc tweaks /
  test additions. Surface byte-identical. Every PR must keep
  `scripts/check-public-api.sh` green without refreezing.

### `1.0.0-rc.1` is the freeze point

This release tag corresponds to the workspace state at slice 66
(2026-05-02). Crates inherit the version from
`workspace.package.version`; bumping this single field promotes
the workspace from `0.1.0` to `1.0.0-rc.1`. RC1 → RC2 → … →
`1.0.0` is the standard pre-1.0 stabilisation cycle; each RC is
a lockstep crates.io publish.

### Public-API baseline is the surface contract

`docs/public-api/<crate>.txt` (20 crates × auto-generated
`cargo-public-api` snapshot) is the *source of truth* for what's
stable. `scripts/check-public-api.sh` runs in CI and fails on
any drift; deliberate changes refreeze via
`scripts/freeze-public-api.sh` and land alongside an ADR
documenting the change.

Pre-1.0 (0.1.x) refreeze was lightweight — RC-unit slices
landed and the baseline updated as a routine. Post-1.0, every
refreeze is an *explicit* MAJOR-bump signal: the next release
cycle carries a new MAJOR version. Reviewers reject silent
refreezes.

### Invariants are the post-compile contract

18 invariants govern code that compiles (CLAUDE.md §"Architecture
invariants"). The CI-enforced subset is checked by 14 scripts plus
the typed-test regression suites listed below:

| 11 (multi-tenant Namespace) | runtime `assert!` in `Namespace::new` (slice 64) |
| 13 (backend isolation) | per-backend `tests/*_isolation.rs` (slice 65) |
| 14 (no shims) | `scripts/check-no-shims.sh` (slice 66) |
| 15 (no silent fallback) | `scripts/check-silent-fallback.sh` (baseline 0 across codecs after slices 60-63) |
| 16 (LLM/operator separation) | `crates/entelix-tools/tests/llm_context_economy.rs` (9 tests) |
| 17 (heuristic policy externalisation) | `scripts/check-magic-constants.sh` |
| 18 (audit channel) | `tests/audit_sink_emits.rs` + per-producer regression tests |
| Lock ordering (CLAUDE.md §"Lock ordering") | `scripts/check-lock-ordering.sh` (workspace `await_holding_lock` / `await_holding_refcell_ref` clippy lints pinned at deny level) |
| Facade feature isolation | `scripts/check-feature-matrix.sh` — every facade feature must compile alone (catches forgotten `entelix-X/feature` pass-through) |
| Workspace dependency hygiene | `scripts/check-dead-deps.sh` — every `[workspace.dependencies]` entry must be inherited by at least one crate |

| (managed-agent shape) | `scripts/check-managed-shape.sh` (ADR-0035 mitigation) |

The other 10 invariants (1, 2, 3, 4, 5, 6, 7, 8, 10, 12) are reviewer-enforced; CLAUDE.md is
authoritative. Adding a new invariant is a MAJOR bump because it
constrains every contributor; removing one is also MAJOR. Both
require a dedicated ADR.

### Companion crate cadence — independent versioning post-1.0

ADR-0008 names the companion-crate pattern: vendor-specific
backends (`entelix-memory-openai`, `entelix-memory-pgvector`,
`entelix-memory-qdrant`, `entelix-graphmemory-pg`,
`entelix-mcp-chatmodel`) ship in the same workspace lockstep
through 1.0.0. Post-1.0, the workspace splits into two release
trains:

- **Core lockstep** — every non-companion crate (12 crates: core,
  runnable, prompt, graph, graph-derive, session, memory,
  persistence, tools, mcp, cloud, policy, otel, server, agents,
  facade) versions in lockstep.
- **Companion crates** — versioned independently. Each tracks
  its upstream vendor SDK / backend version. Compatibility is
  declared in the companion's Cargo.toml — `entelix-core = "1.x"`
  with `^` to allow operators to pick the companion's release
  cadence without forcing core upgrades.

This split lets `entelix-memory-qdrant` ship a `qdrant-client 2.x`
upgrade as `entelix-memory-qdrant 2.0.0` against
`entelix-core 1.x` without forcing every operator onto a
qdrant 2.x backend. Same shape for `entelix-mcp-chatmodel` etc.

### Breaking change policy

Post-1.0, *no* breaking change lands without:

1. **A dedicated ADR** explaining what changes, why, what the
   migration looks like, and what the operator-side cost is.
2. **A MAJOR version bump** in the next release.
3. **A migration guide** in `docs/migrations/` with concrete
   before / after code.
4. **Public-API baseline refreeze** in the same PR, reviewed as
   a deliberate signal.

Pre-1.0 ("processing-as-if-designed-from-the-start" instruction)
allowed in-PR deletes without history. Post-1.0, the four-step
gate is mandatory.

### What is **not** stable

Not every public type is operator-facing API. Specifically:

- **`#[doc(hidden)]` items** — internal but technically `pub`
  for cross-module access. Subject to change without ADR.
- **`*Builder` private fields** — only the public builder
  methods (`with_*`, `add_*`, `set_*`, `register`, `build`) are
  stable. Internal field reordering / renaming is internal.
- **Test fixtures** — anything under `crates/*/tests/` is internal
  to the crate's verification. Operators do not depend on
  fixture types.
- **`docs/public-api/<crate>.txt` itself** — the baseline file
  format is stable but the per-line ordering may shift as
  `cargo-public-api` releases refine its emit. The *set* of
  symbols (line content, ignoring ordering) is what matters.

## Consequences

✅ Operators reading the README know `1.0.0` means semver-strict
post-release: any breaking change lands as `2.0.0` with full
ADR + migration guide + baseline refreeze. Predictable.
✅ Contributors have a single decision tree: change is
additive (MINOR bump, no ADR required for trivial additions)
vs. surface-changing (MAJOR bump, ADR mandatory). Reduces PR
review friction.
✅ Companion crates can ship aggressive vendor SDK upgrades
post-1.0 without forcing the entire ecosystem to follow. Real
deployments often need a specific qdrant / openai SDK version
ahead of the rest.
✅ The 18 invariants become the "post-compile contract" — they
do not change without explicit MAJOR signaling. Operators
reading CLAUDE.md can rely on the invariant set across point
releases.
✅ `docs/public-api/<crate>.txt` becomes the single source of
truth for "what's stable" — operators can grep the file, contributors
diff it. No external reference docs to drift.
❌ Lockstep workspace versioning means an unrelated change in
`entelix-mcp` triggers a release for `entelix-core` etc. Real
cost: more frequent releases, but every release is reviewed by
CI as a unit. Acceptable.
❌ Companion-crate split forces operators to track two version
trains post-1.0. Mitigated by Cargo's resolver — the operator
declares `entelix-mcp-chatmodel = "1"` and `entelix-core = "1"`
and Cargo picks compatible versions.

## Alternatives considered

1. **Per-crate independent versioning from day one** — operators
   choose each crate's version independently. Reality: most
   operators want a single "entelix N.M.K" they upgrade in one
   shot. Per-crate from day one shifts the burden onto every
   downstream Cargo.toml. Rejected for the core; adopted for
   companions where per-vendor cadence is real.
2. **0-ver indefinitely (stay on `0.x`)** — no semver promise,
   in-PR deletes forever. Lighter-weight for contributors but
   blocks adoption — every operator faces the same "is this
   safe to depend on?" question every release. Rejected.
3. **Ship `1.0.0` directly without RC cycle** — every other
   ecosystem expectation is RC1 → RC2 → release. Skipping the
   stabilisation cycle invites public-API churn after the major
   bump. RC cycle gives operators a chance to surface compat
   issues. Rejected.
4. **Make every `pub` symbol part of the contract** — including
   `#[doc(hidden)]` and `Builder` internal fields. Forces every
   contributor edit to consider semver implications. The
   public-API baseline already filters `#[doc(hidden)]` via
   `cargo-public-api --simplified`; the explicit "what's not
   stable" list above is the operator-facing contract. Rejected.
5. **Defer the companion-split decision to post-1.0** — keep
   companions in lockstep for 1.0.x, split at 2.0.0. Operators
   would face a sudden cadence change at the major bump.
   Documenting the split in 1.0 charter prevents surprise.
   Rejected (split policy lands in 1.0).
